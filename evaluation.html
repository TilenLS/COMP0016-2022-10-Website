Critical Evaluation

User Interface & Experience

Through the course of developing the app, we have consistently considered feedback from our client and project supervisors, while also considering the user's perspective.

We keep the user informed when incorrect data such as login credentials or patient data is inputted with appropriate pop-ups. There is also clear contrast for key functionailities for users to easily notice them. We also have easy to understand symbols on key functionailities so that users can be confident they know how to achieve what they want within the app.

The UI is kept simple and straightforward. We ensured that the pages aren't cluttered and crowded, only having key components present. The overall theme is light so that clinicians using the app for long wont strain their eyes.

Functionality

Functionality has been accomplished well in the project as we have fully completed all key functionailities expected of our client. We have also made good progress with the optional requirements too. The implemented functionailities work as intended and perform within expectation.

Stability

Our app is primarily stable with few unexpected issues. However, there are some potential outliers within the expected behaviour of the app. The main one being that the add patient sidebar doesn't automatically close when moving to a page other than the home page.

Efficiency

The app generally performs fast enough to avoid any bad user experience. Running a prediction for a given tooth is actually achieved quite quickly and is not long enough to be noticeable by the user. However, when the tooth models are being uploaded to the database and then retrieved there are issues with effiency. Due to the 3D nature of the data, which for PLY files also contains color, the data is quite complicated and takes time to process. Therefore, it usually takes 5-7 seconds for this process.

All other basic aspects of the app such as opening the app, logging in, adding a patient, loading a STL/PLY file and so on are satisfiably efficient.

Compatibility

Our app is packaged as an EXE file for native desktop Windows applications. This means that it runs on all windows devices. This is due to our client requirements which specifically included that the app must be a Windows native app.

The code itself mostly runs on any device. The only part of the code that will not work anywhere but Windows is for displaying the 3D models on the viewing page. This will only work on Windows due to the use of the Win32GUI Python library which is specific to Windows.

Maintainability

Our app has been strategically seperated into different modules for specific functionalities. This allows other developers to easily understand and work on the code themself. These individual components have been brought together into a main controller module that acts as the API.

Our code is also adequately accompanied with docstrings to ensure that any sections that might be confusing are clarified. This ensures that anyone who might want to use or edit the code can clearly understand what code performs specific functionality.

Project Management

The project has been well managed, finding a good balanced between rigid organisation and flexibility within our timeline. We allocated specific parts of the project to each member while still leaving room to help each other and cooperate.

We used the Gantt chart to ensure we were meeting our deadlines and to plan any meetings with the team, client, or project supervisors we might need.

We also ensured that we had regular progress meetings and online communication to keep each other updated and help if needed. We used this meetings to reasses our strategies and accordingly modify our plans to compensate for any unexpcted changes.

Future Work

Optimization

Currently the AI model we are using has been trained on the very little data we do have access too. Unfortunately, this is mainly due to how sparse PLY tooth models are. In the future, more data will become avaliable through the partnership of the client who has proactively been gathering more tooth scans. With this the AI models will be able to be properly trained to achieve a suitable accuracy for the app to have the intended impact.

Improve AI Model

The model currently used for grading tooth wear probably needs modification to properly suit the data it is being trained on. This is also something thats very hard to develop with the lack of substantial data.

Furthermore, we use TWES 2.0 as our metric for tooth wear evaluation. This consists of grading each sextant of all the teeth seprately and giving an overall score based on those. Due to the lack of data we have only been able to work with and grade such sextants as opposed to the next step which includes the whole teeth. Therefore, in the future a complimentary segemntation model will need to be developed to seperate the teeth in sextants for individual grading.

Improve Decision Support

Currently, our project focused on grading the tooth wear of a patients teeth with which to support dentists. This is core part of the overall algorithm, but the goal is to generate treatment plan suggestions. This will be done in the future by combining these tooth wear grades and the initially inputted patient data as input for the main model which will generate tratment plans based on this input

This will require even more data of dentists' diagnosis and prognosis for each pair of patient tooth models and personal data.

Further Automation

Another curcial development in the future of this project will be to enable automatic self-training of the model(s). With the input of new data the model should automatically re-train to update its weights and reupload them to be used as the current pre-trained model.

With this the software's treatment plan suggestion system will continually grow more accurate as dentists use it.
